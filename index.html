Catalog Core

Items & Catalog

The central entity of the inventory. Every item carries full product metadata, pricing, image, stock thresholds, and supports soft-activation — meaning nothing is ever hard-deleted.

✏️
CRUD Operations
Create, read, update items. Deactivate / reactivate preserves all history. Paginated listing with advanced filters.
🖼
Image Upload
JPEG, PNG, WebP up to 10 MB. Stored in Firebase Storage (unencrypted for direct URL embedding).
💰
Pricing
Cost price and sell price per item. Both toggleable via config. Stored with 3-decimal numeric precision.
🔔
Stock Thresholds
Min and max stock level per item. Automatic low-stock alert trigger when quantity ≤ min threshold.
🏷
SKU & Identifiers
Optional SKU field. Items belong to a section (category) and a unit of measure for display consistency.
🔄
Soft Activation
Items toggle between active / inactive. No data loss — stock entries, barcodes, and movement history retained.

Warehouse Operations

Stock Operations

Six atomic operations — every single one writes an immutable movement record to the audit trail. Stock is tracked at the item × location × lot level.

➕
Adjust
Add or remove arbitrary quantity from a specific item + location (+ optional lot). Supports positive and negative values.
🔀
Transfer
Move stock from one location to another atomically. Both the outbound and inbound movements are recorded.
🔢
Count
Physical count reconciliation — sets stock to exact counted value. Records the delta as a COUNT movement.
🔒
Reserve
Soft-hold stock for a pending commitment. Increases reserved field; available = quantity − reserved.
🔓
Release
Free previously reserved stock back to available pool. Decrements reserved without changing total quantity.
📋
Place Order
Marks stock as on_order (non-blocking). Triggers ORDER_PLACED movement. Reversed via cancel or fulfilled via receive.
IN · OUT · ADJUSTMENT TRANSFER · COUNT RESERVE · RELEASE ORDER_PLACED · ORDER_CANCELLED Every op → movement record

Warehouse Layout

Multi-Location Support

Stock is tracked per-item per-location. Each tenant can define multiple physical storage areas. One location is flagged as the default.

Location Features
🏢
Multiple Locations
Define warehouses, shelves, or any physical area. Each has a name, optional address, and default flag.
📊
Per-Location Stock View
Query available, reserved, and on-order quantities broken down by location for any item.
🔁
Stock Transfers
Atomic transfer between locations — single API call creates both an outbound and inbound movement record.
⭐
Default Location
One location per tenant is flagged as the default — used as fallback in forms and order receiving flows.
Stock Entry Model
🗂
item × location × lot
Each InventoryStockEntry row tracks one item at one location, optionally for one lot.
🔢
Three Quantities
quantity — physical total
reserved — soft-held
available — quantity − reserved
🎯
Unique Constraints
Partial unique index per (tenant, item, location) when no lot. Separate index per (tenant, location, lot) when lot present.

Organisation

Sections & Units of Measure

Two foundational building blocks: sections organise items into coloured categories, units standardise quantities across the system with optional conversion factors.

Sections (Categories)
🎨
Colour & Icon
Each section carries a colour and icon for visual identification across the UI and alert routing.
🔢
Sort Order & Item Count
Sections carry a sort_order for custom ordering. The list endpoint returns live item_count per section.
👥
Section Subscribers
Users can subscribe to specific sections to receive low-stock and expiry alerts only for items in those sections.
Units of Measure
⚖️
Custom Units
Define any unit — kg, pieces, litre, box. Each has a name, symbol, and category (weight, volume, count…).
🔄
Conversion Factors
Optional base_unit_id + conversion_factor enables consistent cross-unit quantity comparisons.
🏷
Display Symbol
The unit symbol is snapshotted onto requisition items at creation for immutable display even if the unit later changes.

📷
Scanning

Barcode Scanning

Items support multiple barcodes with one flagged as primary. Scanning works via device camera or an external USB/Bluetooth scanner.

📱
Camera Scanning
Real-time camera feed decodes barcodes directly in the browser — no native app required.
🔌
External Scanner
USB / Bluetooth barcode guns are supported as keyboard-input devices — plug and use immediately.
🏷
Multi-Barcode per Item
Each item can carry any number of barcodes (EAN, QR, etc.). One is flagged as the primary barcode.
🔎
Barcode Lookup
A dedicated lookup endpoint resolves any registered barcode string to its full item record instantly.
➕
Add / Remove
Barcodes can be added or removed from an item at any time. Deletion requires the specific barcode ID.
⭐
Primary Flag
One barcode per item carries the is_primary flag, used as the default scan target and export identifier.

Procurement

Supplier Management

Manage external suppliers and link them to specific items with per-supplier pricing. Supplier records feed directly into the purchase order system.

Supplier Records
🏢
Full Contact Info
Name (unique per tenant), email, phone, company identifier, notes, and active/inactive toggle.
🔍
Search & Filter
Paginated list with search and optional include-inactive flag for comprehensive supplier visibility.
🔗
Item Count
Each supplier record returns a live item_count — how many inventory items are linked to this supplier.
Item–Supplier Links (Pricing)
💱
Per-Supplier Pricing
Each link stores supplier_price per base unit, supplier_item_code, and the supplier's own unit + quantity.
📝
Supplier Item Code
Maps your internal item to the supplier's own catalogue reference code for accurate order lines.
⚖️
Unit Normalisation
supplier_unit_quantity normalises supplier pack sizes to your base unit for consistent comparison and ordering.

Internal Requisitions

Employees request items from stock. Approvers review, then fulfill from specific locations using FEFO-aware allocation — automatically consuming the earliest-expiring lots first.

📝

Fulfillment Engine
🏭
Stock Availability Check
Before fulfilling, the system returns per-item availability broken down by location and lot, with suggested allocations.
📅
FEFO-Aware
Suggested allocations automatically prioritise lots with the earliest expiration dates first across all locations.
⚡
Atomic Fulfilment
The entire fulfill operation runs inside one database transaction — all locations consumed or nothing.
Approval Config
⚙️

Tenant admins configure a list of user IDs as requisition approvers. Empty list = open approval policy.
🔀

When requisitionApprovalRequired is false, submit moves the requisition directly to ready-to-fulfill state.
🔔
In-App Notifications
Approvers notified on submission. Requester notified on fulfillment, rejection, or cancellation.

Scheduled Monitoring

Expiry Alerts

A daily cron-driven system that identifies lots nearing or past expiration, then delivers batched digest notifications. Firestore indexing ensures only tenants with expiring lots are checked each day.

⏰
Daily Cron
QStash schedule
Managed by ExpiryScheduleService
→
🗂
Firestore Index
earliest_expiration per tenant
Skips tenants with no upcoming expiries
→
🔍
Query Expiring Lots
per-tenant batch
Finds lots expiring within alert window
→
📊
Build Digest
grouped by item / lot
Single message per recipient
→
📨
Dispatch
email + SMS + in-app
Via Messaging cellabon channels
⚡
Efficient Targeting
ExpiryTrackingService maintains a Firestore index of earliest_expiration per tenant — the cron only hits tenants that need it.
📬
Digest Format
Recipients receive one batched digest message listing all expiring lots — not one message per lot.
🔗
Cross-Cellabon
Expiry alerts are enabled via the Messaging cellabon's enableExpiryAlerts toggle, mirroring the low-stock alert pattern.
